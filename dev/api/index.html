API reference

This page summarizes the public API exported by CodeEdit.jl.

Handles

  • Handle: create a handle to the block containing a file location or method.
  • eof_handle: create a handle to the end of a file.
  • handles: collect handles for blocks in files.
  • reindex: update existing handles after files changed outside CodeEdit.jl.

Searching

  • search: search handles, files, stacktraces, or exceptions.

Edits

  • AbstractEdit: abstract supertype for edit values.
  • Replace: replace a block.
  • Delete: delete a block.
  • InsertBefore: insert code before a block.
  • InsertAfter: insert code after a block.
  • CreateFile: create a new file.
  • MoveFile: move or rename a file.
  • DeleteFile: delete a file.
  • Combine: combine edits into one planned edit.
  • apply!: apply an edit through an explicit version-control specification.
  • displayed!: mark an edit as displayed.

The edit1 * edit2 operator is shorthand for Combine(edit1, edit2). Chaining * appends edits to a combined edit.

Combined edits are planned and validated as a unit, but applying a combined edit that touches multiple files is best-effort at the filesystem level and can partially apply if a later filesystem operation fails.

Version control

  • VersionControl: describe a git repository and default apply! keyword arguments.
  • GitVersionControl: convenience constructor for git-backed editing.
  • NoVersionControl: explicitly apply edits without version control.
  • ApplyResult: successful result returned by apply!.

apply!(repo, edit, message) applies an edit, stages affected paths, and creates a git commit with message.

apply!(repo, edit; default_message="...") uses a default commit message supplied either in the call or in the VersionControl object.

apply!(NoVersionControl(require_view=true), edit) applies without git while still requiring a displayed review.

Display, printing, and string(edit) store the exact plan that was shown. When require_view=true, apply! replans the edit and rejects it if the current plan differs from the displayed plan.

Convenience functions

  • filepath: return the file path for a handle.
  • lines: return the line range for a handle.
  • docstring: extract an attached docstring.
  • is_valid: test whether a handle or edit is valid.
  • string(handle): return the block text for a handle.
  • string(edit): return the displayed edit plan and mark the edit as displayed.
  • display(handle): show a handle header and source block.
  • display(edit): show the edit plan and mark the edit as displayed.
  • occursin(handle, trace): test whether a handle's source location occurs in a stacktrace-like object.

CodeEdit.Handle — Type
Handle(path, line, pos=1; parse_as=:auto)
Handle(method)

Reference to a parsed source/text block.

The path-based constructor returns a handle to the block containing (line, pos), or to the next block after that location. The method-based constructor returns a handle to a method definition when source information is available.

CodeEdit.handles — Function
handles(path; includes=false, parse_as=:auto)
handles(paths; includes=false, parse_as=:auto)
handles(root, pattern; includes=false, parse_as=:auto)

Return handles for all parsed blocks in one file, a collection of files, or files under root matching pattern.

If includes=true, statically resolvable Julia include(...) paths are followed recursively. parse_as may be :auto, :julia, or :text.

CodeEdit.reindex — Function
reindex(path)
reindex()

Reparse cached files and conservatively preserve handles for uniquely matched blocks.

The path form reindexes one cached file. The zero-argument form reindexes all cached files that still exist.

CodeEdit.Combine — Type
Combine(edits::AbstractEdit...)
Combine(edits::AbstractVector{<:AbstractEdit})

Edit that combines multiple edits into one planned operation.

Combined edits are interpreted in order and validated as a unit. Applying a combined edit that touches multiple files is best-effort at the filesystem level, so a later filesystem failure can leave earlier operations applied.

CodeEdit.apply! — Function
apply!(edit::AbstractEdit)

Reject edits applied without an explicit version-control specification.

Use apply!(NoVersionControl(require_view=true), edit) for behavior equivalent to the old apply!(edit), or apply!(VersionControl(path), edit, message) to apply and commit an edit in a git repository.

source
CodeEdit.displayed! — Function

Set or clear the displayed marker for an edit.

When displayed is true, this compiles and validates the current edit plan and stores its fingerprint as the plan approved for application. It does not print the diff. Use this only when intentionally bypassing visible review.
